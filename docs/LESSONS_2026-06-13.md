# 세션에서 배운 점 — 2026-06-13 (토큰 5h% 계정 라벨 오표기 — 게이트의 fail-open 을 표시층에도)

> "무엇을 만들었나"가 아니라 **다음에 시간을 아껴 줄 비자명한 운영·검증 교훈**.
> 구체 명령·파일·라인으로 남긴다. 배경: 사용자 신고 — 토큰 팝업의
> "Account (/usage): me@woojinkim.org" 와 하단 토큰 표시 "0%/5h used
> wo…@nexongames.co.kr" 가 **서로 다른 계정**을 가리킴. 수정: `_tok5h_pct` 계정
> 불일치 fail-open(servermixin.py:1334) + 회귀 테스트.

## 1. 같은 데이터의 두 소비처는 같은 불변식을 써야 한다 — 게이트엔 있고 표시엔 없던 검사

- 5h% 실측은 **그림자 /usage 프로브 세션**의 계정(`_usage["account"]`)에서 나오고,
  하단 상태줄은 그 %에 **포그라운드 패널**의 계정(`_claude_account`)을 라벨로 붙였다.
  두 계정이 다르면(프로브가 다른 로그인으로 떠서 측정) "다른 계정의 한도를 이 패널
  계정으로 표기"하는 오표기가 된다. 팝업은 측정 계정을 정직하게("Account (/usage):")
  보여서 두 표면이 어긋났다.
- **핵심 교훈**: 정확히 이 계정-불일치 fail-open 은 **게이트엔 이미 있었다** —
  `_usage_gate_pcts`(servermixin.py:1804)는 "프로브 계정과 패널 계정이 둘 다 알려져
  있고 다르면 None(다른 계정 한도로 이 패널을 막지 않음)". 그런데 같은 `_usage` 를
  소비하는 **표시 경로** `_tok5h_pct`(servermixin.py:1326)는 그 검사를 건너뛰고
  `_usage["session"]["pct"]` 를 무조건 반환했다. **한 데이터(`self._usage`)에 두
  소비처(게이트·표시)가 있는데 한쪽만 불변식을 지킨 것이 버그의 정체.**
- 일반화: "X 를 적용할지 말지"의 판단(여기선 계정 일치)이 한 소비처에 있으면,
  같은 X 를 읽는 **모든** 소비처를 grep 해 같은 게이트를 거치는지 확인하라. 새 표면을
  추가할 때가 아니라 **기존 표면이 조용히 가정을 안 따를 때** 어긋난다(2026-06-12 §7의
  "한 표면만 가정을 안 따름"의 재발 — 그땐 단일계정 귀속, 이번엔 계정 일치 fail-open).

## 2. fail-open 의 "표시판" vs "게이트판" 은 신선도 처리가 갈린다 — 통째 재사용 금지

- `_tok5h_pct` 에서 `_usage_gate_pcts` 를 그대로 호출하고 싶었지만 **안 했다**:
  게이트판은 `_usage_fresh()`(stale 이면 None)까지 묶지만, **표시는 stale 실측도
  보여야 한다**(팝업이 경과 시간 라벨과 함께 stale 을 표시하는 것과 일관 —
  usage_age_sec). 그래서 fail-open 조건 중 **계정-불일치 부분만** 떼어 표시층에
  복제했다. 같은 원칙이라도 "개입 보류(게이트)"와 "라벨 정직성(표시)"은 신선도
  요구가 달라, 공유 헬퍼로 묶으면 표시가 stale 을 잃는다.
- 교훈: fail-open 조건이 ①데이터 부재 ②stale ③의미 불일치(계정) 처럼 다축이면,
  소비처마다 **필요한 축만** 켜라. 통짜 헬퍼는 가장 보수적인 소비처에 나머지를
  강제한다.

## 3. "두 화면이 다른 값" 신고는 각 값의 **데이터 출처를 grep 으로 역추적**부터

- UI 좌표(팝업 vs 하단바)로 추측하지 말고 각 숫자가 읽는 필드를 끝까지 따라갔다:
  팝업 계정 = `usage["account"]`(usage_bar_lines, clientscreens.py) ← `server._usage`,
  하단바 계정 = `status.claude_account`(render_segs, clientstatus.py:200) ←
  `ap._claude_account`(서버 server_status). **출처가 갈리는 지점**(프로브 세션 vs
  포그라운드 패널)이 곧 버그의 원인이었다. 두 출처가 만나는 5h%(`tok5h_pct`)가
  결합점이라 거기를 고쳤다.
- 절차: `grep -rn "<표시 문자열의 i18n 키>"` → 렌더 함수 → 읽는 status 필드 →
  `absorb`/`update_status` → server `_status_msg` 키 → 서버측 산출 함수. 표시 두 개면
  **두 체인을 끝까지 따로** 그려 분기점을 찾는다.

## 4. 서버측 표시 로직 수정은 라이브 반영에 restart 필요 — 드라이런 먼저

- `_tok5h_pct` 는 서버 코드라 편집만으론 라이브 세션에 안 먹는다(restart-dryrun-first
  프로토콜). 테스트(593 green)는 즉시 검증되지만 사용자 화면 반영은 `restart-server`
  (work-preserving 재-exec) + 사용자 확인이 별도. 표시 버그는 "테스트 통과 = 끝"이
  아니다 — 실제 표면 갱신은 restart 뒤다.
