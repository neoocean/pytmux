## 1. "데이터가 부정확하다" 신고는 라이브 DB GROUP BY 한 방부터 — 원인이 셋으로 분해됐다

- "엉뚱한 계정·시간 뷰 부정확·날짜 누락" 신고를 UI 버그로 묶어 추측하지 말 것.
  `sqlite3 …/claude-code/db/claude-tokens.db "SELECT account, COUNT(*), SUM(tokens)
  FROM usage GROUP BY account"` 한 줄이 세 원인을 즉시 분리했다:
  ① **데이터 오염**(가짜 계정 "Running 1 shell command" 2행 — 스크랩 오탐),
  ② **모델 부재**(unknown 86% — 계정 라벨이 /status 에만 떠 미식별 다수),
  ③ **표현 문제**(계정행+시간행 혼합 표 — 필터 시 미식별 날짜가 통째로 사라져 보임).
  각각 다른 층(claude.py 검출 / 표시층 귀속 / 화면 구조)의 수정이라 CL 도 갈라졌다.

## 2. 휴리스틱 스크랩 신호는 **형태 제약이 있는 것만** 신뢰 — 자유 텍스트 패턴은 구조적으로 오탐

- `_ORG_RE`(`account|team|… : <아무 구절>`)는 Claude 의 도구 출력 산문
  ("…Running 1 shell command")을 계정으로 잡았다. 자유 텍스트를 받는 캡처 그룹은
  화면에 무엇이 뜰지 통제할 수 없어 **오탐을 막을 방법 자체가 없다**.
  이메일(`local@domain`)처럼 형태 제약이 강한 신호만 남기는 게 답(58538) —
  "잘못된 계정 표시보다 unknown" 원칙(2026-06-07)의 연장.
- 오염 데이터 정정은 **비파괴 스키마 마이그레이션** 패턴 재사용: v3 의
  `usage_dup_archive` 선례대로 v4 도 원값을 `usage_acct_fixlog`(rowid·원 계정)에
  보존하고 UPDATE — 포렌식·복구 가능. 판별식은 "신뢰 신호가 전부 이메일이므로
  '@' 미포함 = 오탐"(`usagedb._migrate_v4_nonemail_accounts`).

## 3. % 지표는 **모든 표면에서 같은 방향** — 라벨 추가로는 부족했다(신고 2회의 교훈)

- 같은 실측(세션 15% 사용)을 팝업은 사용률("15%"), footer 는 잔여율("85%/5h 남음")로
  보여 사용자는 두 번이나 "값이 다르다"고 신고했다. 1차 대응(58559: 막대에
  '사용' 라벨)으로도 혼동이 남았고, 결국 **방향 자체 통일**(58568: footer 도
  "{pct}%/5h 사용")이 답이었다.
- 일반화: 보수 관계(사용=100-잔여)는 설명해도 체감 불일치가 사라지지 않는다.
  새 % 표시를 추가할 땐 **같은 수치를 보이는 다른 표면들을 먼저 인벤토리**하고
  방향·라벨을 맞춰라(이번엔 팝업 막대·usage-panel·footer·Claude /usage 원문 4곳).

## 4. 렌더 "이상" 신고는 REC 캡처 이스케이프 해독이 권위 — 이번에도 코드 무변경으로 종결

- "빈 프롬프트가 두 줄" 신고 → 캡처 원시 바이트를 해독하니 **Claude Code 자신의
  빈 컴포저 스타일**(회색 보더줄 + `❯` 입력줄, 입력 시 보더 없는 음영 한 줄로
  재렌더)이었다. pytmux 는 충실 렌더 중 — 고칠 게 없었다.
  [claude-scrape-false-positives 프로토콜("캡처 원시바이트부터")]의 재확인.
- 절차 디테일(다음에 그대로 재사용):
  - **contiguous 문자열 검색은 실패할 수 있다** — Claude 는 스피너/텍스트를 글자
    단위 증분 렌더해 "Crunched" 같은 단어가 통짜 바이트로 안 적힌다. 못 찾으면
    `[2J`(전체 리페인트) 지점을 잡아 거기부터 디코드하라.
  - 프레임 복원은 `pytmuxlib.replay.replay(seg, cols, rows)` — **cols 는 보더줄
    `─`×N 길이로, rows 는 프레임 끝 커서 파킹 `ESC[N;1H` 로 역산**한다(이번엔
    50×20). 리사이즈가 낀 캡처를 처음부터 통재생하면 줄이 뒤섞인다 — 마지막
    `[2J` 이후만 재생.
  - 스크린샷의 패널이 **어느 탭/패널인지부터 확정**하라(본문 내용으로 식별).
    엉뚱한 캡처(p2)를 먼저 뒤져 시간을 버렸다 — 신고 화면의 본문 문구로
    `captures/*/…_p<N>.log` 를 고르는 게 첫 단계.

## 5. `runner | tail && submit` 은 게이트가 아니다 — 파이프라인 종료코드는 tail 의 것

- CL 58556 제출 체인 `tests/run.py … 2>&1 | tail -2 && p4 submit` 에서 스위트가
  1건 TIMEOUT(실패, run.py exit 1)이었는데도 **submit 이 진행**됐다 — `&&` 가
  본 종료코드는 파이프 마지막 명령(tail=0)이다. zsh/bash 기본값에서 파이프는
  마지막 명령의 상태만 반환한다.
- 대응: 게이트가 목적이면 `set -o pipefail` 을 켜거나, 러너를 **파이프 없이**
  돌리고 결과를 따로 grep 하라. (이번엔 사후 재실행으로 flake 임을 확인해 실해는
  없었지만 — `test_status_account_full_when_wide_alias_when_narrow` 1회 90s
  TIMEOUT 후 2회 연속 통과, 부하성 의심 — 게이트가 뚫린 건 사실이다.)

## 6. 같은 파일을 공유하는 두 변경의 CL 분리 — 한쪽 hunk 임시 되돌리기

- p4 는 파일 단위 오픈이라 `tests/test_client.py` 에 두 CL 분량의 편집이 섞이면
  못 가른다. **나중 CL 의 hunk 만 정확히 원문으로 되돌리고(파이썬 replace 스크립트)
  먼저 CL 을 제출 → hunk 재적용 → 다음 CL** 로 각 CL 을 자기완결로 유지했다
  (58556↔58559). 되돌린 상태로도 스위트가 green 인지 확인하고 제출할 것.
- 부속: 파일이 없는 빈 pending CL 은 `p4 changes -s pending <path>` 에 안 보인다 —
  `p4 change -i` 직후 출력된 번호를 그대로 쓰라.

## 7. 표시층 가정(단일 계정 귀속)은 **이미 쓰는 표면과 같은 가정**으로 정렬하라
