# 세션에서 배운 점 — 2026-06-10 (반쯤 통합된 플러그인 WIP 크래시, 훅↔호출부 동행 원칙)

> "무엇을 만들었나"가 아니라 **다음에 시간을 아껴 줄 비자명한 운영·구현 교훈**.
> 구체 명령·파일·라인으로 남긴다. 어제분은 [LESSONS_2026-06-09](LESSONS_2026-06-09.md).

## 1. 레지스트리 훅과 호출부는 **한 CL로 같이** 움직여야 한다

- 이번 라이브 크래시: `AttributeError: 'Registry' object has no attribute 'client_overlay'`.
- 원인은 **반쯤 통합된 플러그인 WIP**였다. 병렬 세션이 clock/calendar 를 추출하며
  `client.py` 에 `self.plugins.client_overlay/client_tick/client_close_overlay` **호출부**를
  넣었는데(미커밋), 짝이 되는 `plugins/__init__.py` **Registry 훅**은 빠져 있었다.
- 왜 빠졌나: 그 훅은 워킹트리에만 있던 미커밋 변경이었고, 어느 `p4 sync` 가
  `plugins/__init__.py` 를 **depot head(훅 없는 버전)** 로 되돌리는 동안 `client.py` 의
  호출부 WIP 는 (opened 가 아니라) 그대로 남아 **호출부만 살아남은 불일치**가 됐다.
- **원칙**: 코어의 `plugins.<hook>(...)` 호출부와 그 훅의 Registry 정의는 **분리되면
  즉시 크래시**한다(덕 타이핑이 아니라 Registry 메서드 자체가 없으면 AttributeError).
  새 훅을 도입할 땐 **호출부 + Registry 정의를 같은 변경/같은 CL** 로 묶어라. 한쪽만
  sync/revert 되는 창에 노출시키지 말 것.

## 2. "내 변경 vs 병렬 WIP" 3초 분리법

공유 워크스페이스([[shared-workspace-parallel-wip]])에서 크래시를 만나면:

- `grep -rn "<없는속성>" pytmuxlib/*.py` 로 **호출부 위치**부터 특정(여기선
  client.py:1021/1034/1329).
- **호출부가 depot 에 있나** 확인: `p4 print -q //…/client.py#head | grep client_overlay`
  → 비면 그 호출부는 **누군가의 미커밋 WIP**(내 것 아닐 수 있음).
- `p4 opened` 가 **비어 있어도** 워킹트리는 depot 과 다를 수 있다(이 워크스페이스는
  reconcile 형). "opened 없음 = 변경 없음" 아님.
- 내 단독 변경인지 최종 확인: `diff <(p4 print -q //…/file#head) file` 로 **내 hunk만**
  찍히는지(병렬 마커 0건) 본다. submit 전 이 한 줄이 격리 증명.

## 3. 깨진 호출부의 최소·안전 수정 = **no-op 훅 추가**

- 호출부(병렬 WIP)를 지우는 대신, **빠진 Registry 훅을 서버 훅과 동일한 no-op 패턴**으로
  추가해 깨진 통합을 메웠다(`plugins/__init__.py` 의 `client_overlay`/`client_tick`/
  `client_close_overlay`). 플러그인 부재 시 no-op/False → `delete-to-disable` 유지,
  존재 시 각 plugin 동명 메서드로 위임.
- 이렇게 하면 **남의 WIP 를 파괴하지 않고**(호출부 보존) 앱이 다시 뜬다. 동시에 그
  훅은 병렬 세션이 어차피 필요로 하던 다리라 작업을 **되돌리지 않고 완성**한다.

## 4. 남의 기능 조각을 **단독 서브밋하지 않는다**

- 수정은 깨끗했지만(내 단독, 병렬 마커 0), 이 훅은 **병렬 세션 clock/calendar 기능의
  일부**다(호출부 `client.py` + plugins/{clock,calendar}/ 가 그들 미커밋 WIP).
- 훅만 단독 submit 하면 병렬 세션이 자기 `client.py`+`plugins/__init__.py` 를 올릴 때
  **resolve 충돌**이 나고 기능이 두 CL 로 쪼개진다. → **결정: 서브밋하지 않고 워킹트리
  수정만 유지**, 기능 소유 세션이 호출부+훅을 한 CL 로 올리게 둔다(사용자 승인).
- 일반화: 라이브 크래시는 **워킹트리 수정으로 즉시 해소**하되, 그 변경이 다른 세션의
  미완 기능 영역이면 **게시(submit)는 소유자에게** 맡긴다. "고침"과 "게시"는 별개 결정.

## 5. 검증: client.py 회귀망은 **412 테스트**뿐

- driver smoke 는 소켓 클라라 `client.py` 앱 코드를 실행하지 않는다 → 이 크래시를 못 잡는다.
- `python3 tests/run.py` 의 **412개**가 PytmuxApp 인스턴스화/훅 경로를 직접 친다.
  수정 후 4개 플러그인(calendar/claude-code/clock/ncd) 정상 로드 + **412 passed, 0 failed**
  로 회귀 없음을 확인. 더미 app 으로 훅 3종이 no-op 반환하는지도 즉석 검증.

## 6. 코어/plugins 게시는 **p4 submit 만**, git push 아님

- 이 repo 의 git 미러(github.com/neoocean/pytmux)는 **docs+bench 전용**이다. `pytmuxlib/`
  코어와 `plugins/` 는 **git 에 미러되지 않는다**(ncd·claude-code 가 여러 CL 동안 git
  untracked 였던 게 증거).
- 그래서 사용자의 "서브밋·커밋푸시" 지시라도 **코드/플러그인은 p4 submit 으로 끝**이고
  git push 대상이 아니다. clock/calendar(7파일)는 **p4 57879 단독**으로 게시, git push 없음.
- 반대로 **이 교훈 문서·벤치 산출물은 git 도 푸시** 대상. "무엇을 어디에 올리나"는
  파일이 **코드냐 docs/bench 냐**로 갈린다([[publish-workflow-p4-and-git]]).

## 7. 공유 워크스페이스 서브밋 위생: **번호 CL + 파일 단위 reconcile**

- 병렬 세션이 같은 트리를 동시에 만지므로 **default changelist 금지** — 항상 번호 CL
  (`p4 change -i` → `p4 submit -c NNNNN`)로 내 파일만 묶는다.
- `p4 reconcile <특정파일들>` 만 — **bulk reconcile 금지**. 디렉토리 통째 reconcile 하면
  남의 미커밋 WIP 까지 내 CL 로 빨려 들어간다. 57879 는 7파일을 **개별 reconcile** 했다.
- 게시 직전 격리 증명 한 줄: `diff <(p4 print -q //…/file#head) file` 로 내 hunk 만
  찍히는지(병렬 마커 0) 확인([[shared-workspace-parallel-wip]]).

## 8. `p4 reconcile/submit` 이 Bash **샌드박스에 가려진다**

- 샌드박스(서버 쓰기 차단) 안에서 `p4 reconcile`/`submit` 은 실제 파일이 있어도
  "no file(s) to reconcile" 로 **조용히 빈손** 반환한다(서버 write 가 막혀서).
- 해결: 이 둘만 **`dangerouslyDisableSandbox: true`** 로 실행. `-n` 프리뷰는 샌드박스에서도
  보이므로 "프리뷰엔 뜨는데 reconcile 은 빈손"이면 샌드박스부터 의심.

## 9. 클라 플러그인 설치 = **attach_client 인스턴스 클로저** (동적 믹스인 불가)

- `PytmuxApp` 은 `build_client_app()` 팩토리 **안의 중첩 클래스**(메서드가 8-스페이스
  들여쓰기)라, 서버처럼 **동적 베이스 믹스인 합성을 못 한다**(클래스가 런타임 클로저에 갇힘).
- 그래서 클라 플러그인은 상태·메서드를 **`attach_client(app)` 에서 인스턴스에 직접 설치**
  한다(예: `app.clock_panes`, `app.toggle_clock` — ncd 의 `app.request_nc_list` 선례).
  `attach_client` 는 client.py:96(상태 init 직전)에 1회 호출된다.
- 코어는 설치된 메서드를 **getattr 가드**로 부른다: `fn = getattr(self,"toggle_clock",None);
  fn and fn(act)`. 플러그인 삭제 시 → no-op(delete-to-disable). 선례: `open_model_config`.

## 10. 플러그인 내부 import 는 **절대경로** (상대 `..` 함정)

- `plugins/clock/__init__.py` 에서 `from ..clientutil import theme_color` 는 **틀린다** —
  `pytmuxlib.plugins.clientutil`(없음)로 해석돼 8개 테스트가 `No module named` 로 깨졌다.
- **절대경로**로: `from pytmuxlib.clientutil import theme_color` /

## 11. 코어에 **얇은 위임자**를 남겨야 하는 호출부가 있다

- 기능을 통째로 플러그인에 옮겨도 **코어가 무조건 부르는 진입점**은 코어에 얇게 남겨야 한다.
- 사례: `clientwidgets.py:264` 가 **매 패널 클릭마다** `self.app._close_overlay(op["id"])`
  를 부른다. `_close_overlay` 를 플러그인으로 옮기면 오버레이 무관한 일반 패널 클릭마다
  `AttributeError`. → client.py:1015 에 `client_close_overlay` 훅으로 라우팅하는 **얇은
  델리게이터**(`_close_overlay`/`_close_active_overlay`/`_clock_tick`)를 코어에 유지.
- 일반화: "이 메서드를 호출하는 게 **플러그인 코드냐 코어 코드냐**"를 보라. 코어가 부르면
  코어에 (위임만 하는) 껍데기가 남아야 하고, 그 껍데기가 빈 Registry 에서도 안전해야 한다.
