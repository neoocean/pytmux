# 세션에서 배운 점 — 2026-06-12 (owned-ConPTY 기본 백엔드 flip 검증 방법론)

> "무엇을 만들었나"가 아니라 **다음에 시간을 아껴 줄 비자명한 운영·검증 교훈**.
> 구체 명령·파일·라인으로 남긴다. 직전분은 [LESSONS_2026-06-10](LESSONS_2026-06-10.md).
> 배경: §1.1② owned-ConPTY 를 Windows 기본 백엔드로 flip(CL 58503) + 실 Claude 패널
> 라이브 검증. 관련 [WINDOWS_TESTING.md](WINDOWS_TESTING.md)·[IMPROVEMENT_OPPORTUNITIES.md](IMPROVEMENT_OPPORTUNITIES.md) §1.1.

## 1. 멀티바이트 손상은 **렌더된 화면**으로 판정한다 — raw 바이트 스트림 FFFD 카운트가 아니라

- 250KB CJK 버스트 중 raw VT 스트림엔 수백 개 U+FFFD 가 섞이지만 대부분 **다음 리페인트에
  스크롤오프**돼 안정 화면엔 안 남는다. raw 스트림 FFFD 카운트로 "손상 N개" 라고 보고하면
  사용자가 실제로 보는 화면과 어긋난다.
- **방법**: 드라이버로 안정화 후 `d.panes[active]` 의 셀 문자열에서만 `"�"` 를 세라.
  raw conout 바이트가 아니라 **pyte 가 그린 최종 셀**이 권위값이다.
  (이 세션 측정: owned 안정 화면 9개 — 전부 와이드문자 한 칸씩, 산발.)
- 이건 [[pytmux-1-1-multibyte-winpty-corruption]] 에도 박힌 핵심 교훈의 재확인이다.

## 2. cp949 콘솔 mojibake ≠ PTY-계층 손상 — 글리프 검증엔 `chcp 65001` 필수

- 한국어 Windows 콘솔(코드페이지 949)에서 **UTF-8 파일을 `type`** 하면 cmd 가 파일 바이트를
  그대로 콘솔에 덤프 → conhost 가 그 UTF-8 바이트를 **cp949 로 오해석**해 한글이 와장창
  깨져 보인다. 이건 **콘솔-계층** 아티팩트지 PTY/디코드 계층 손상이 아니다.
- 결정적 신호: 그 깨진 화면에서도 **U+FFFD = 0**. mojibake(잘못된 글리프)와 corruption
  (U+FFFD 치환)은 **다른 현상**이다 — FFFD 0 이면 바이트→str 디코드 경로는 깨끗한 것.
- **글리프 자체의 정확성**(와이드문자 정렬·보더)을 검증하려면 버스트 전에 `chcp 65001`
  (콘솔을 UTF-8로) 을 먼저 보내라. 그래야 `한글ＣＪＫ符号テスト…` 가 올바른 글리프로
  렌더돼 보더 정렬을 진짜로 볼 수 있다. (Claude·pwsh 등은 자기 출력을 UTF-8로 내므로 이
  함정이 없다 — 그래서 실 Claude 패널은 chcp 없이도 깨끗했다.)

## 3. 기본값 flip 의 게이트 = **동일 버스트로 현행 기본값과 A/B** ("완벽"이 아니라 "현행 이상")

- owned 단독으로 "9 FFFD 남음" 만 보면 회귀처럼 보인다. **같은 250KB 버스트를 기본
  pywinpty 로도** 돌리니 **FFFD 9개·완전히 같은 위치**(`ふり���な` on line 01261) →
  잔여 손상은 **번들 OpenConsole 공통**이지 owned 회귀가 아님이 증명됐다.
- 일반화: 백엔드/구현 교체의 합격선은 "**현행 기본값보다 나쁘지 않다**"지 "결함 0" 이
  아니다. 신·구를 **동일 입력**으로 나란히 측정해 차이만 본다. A/B 없이 신 코드의 절대
  수치만 보면 멀쩡한 전환을 회귀로 오판한다.
- 측정 격리: 드라이버 출력 폴더를 `pytmux-verify-{backend}` 로 백엔드별 분리(`PYTMUX_PTY_BACKEND`
  값으로 suffix)해 owned/default 캡처가 서로 덮어쓰지 않게 했다.

## 4. 헤드리스 텍스트 스크린샷은 **FFFD·보더 어긋남을 충분히 잡는다** (픽셀 없이도)

- "픽셀 캡처가 안 되니 사람 눈만이 게이트" 는 절반만 맞다. 드라이버의 합성 셀(`screen_text`)에는
  **U+FFFD 와 각 행의 `│` 컬럼 위치**가 그대로 들어 있다 → 손상 카운트·우측 보더 정렬은
  텍스트로 정량 검증 가능.
- 사람 눈이 진짜로 필요한 잔여 1% = **실제 픽셀 렌더**(폰트 합자·터미널 자체 와이드문자 셀
  폭 처리 등). corruption/정렬 판정엔 헤드리스로 충분하므로, "검증 불가" 라고 미루지 말고
  **정량 가능한 부분은 드라이버로 먼저 끝내라**. 이 세션은 그렇게 flip 게이트의 대부분을
  헤드리스로 닫았다.

## 5. flip 자체는 **술어 1줄 반전 + env 토글을 롤백 경로로 보존**

- `pty_backend.spawn()` Windows 분기: `if choice in ("owned","conpty")` →
  **`if choice not in ("pywinpty","winpty")`** 로 반전하면 미설정/빈값/`owned`/`conpty`
  전부 owned, `pywinpty|winpty` 만 `_WinPty` 강제(=롤백 토글). conpty 미지원·spawn 예외 시
  `_WinPty` 자동 폴백은 그대로 안전망으로 남긴다.
- 핵심: **기존 opt-in env 변수를 버리지 말고 롤백 토글로 재활용**하라. 기본만 바뀌고
  되돌릴 1줄 경로(`PYTMUX_PTY_BACKEND=pywinpty`)가 환경변수 하나로 살아 있으면 flip 위험이
  최소화된다. 테스트도 같은 CL 에서 의미 반전(`test_spawn_selects_backend`: 미설정/빈값/
  명시 owned 3건→owned, pywinpty/winpty 2건→winpty)을 함께 고쳐 stale 을 안 남긴다.

## 6. PowerShell here-string → `p4 change -i` 는 **BOM 트랩** (재발)

- `@'...'@ | p4 change -i` 는 **UTF-8 BOM** 을 앞에 붙여 `Error: Unknown field name '﻿Change'`
  로 죽는다(보이지 않는 `﻿` 가 `Change` 앞에 붙음). [[windows-workspace-perforce-only]] 기록의
  실제 재발.
- **해결**: CL 스펙은 **Bash heredoc**(`cat > /tmp/spec <<'EOF' … EOF`, UTF-8 no BOM)으로
  쓰고 `p4 change -i < /tmp/spec`. 이 워크스페이스의 p4 텍스트 입력은 전부 BOM 없는 경로로.

## 7. cp949 박스에서 CJK 를 찍는 검증 스크립트는 **stdout 재구성** 필수

- 검증 스크립트가 CJK/`≈`/이모지를 `print` 하면 `UnicodeEncodeError: 'cp949' codec` 으로
  죽는다(콘솔 기본 코드페이지 949).
- 스크립트 상단에 `sys.stdout.reconfigure(encoding="utf-8", errors="replace")` (stderr 도) +
  실행 시 `PYTHONIOENCODING=utf-8`. 그래야 캡처한 CJK 화면을 콘솔로 그대로 덤프해 눈으로
  확인할 수 있다.
