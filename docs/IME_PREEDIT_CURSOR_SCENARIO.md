# 한글(IME) 조합 글자가 패널 아웃라인에 잔상으로 뜨는 문제 — 동작 시나리오

> **상태**: ✅ **구현됨**(방식 A — Textual 스크린 커서 API). `_composite` 가 매 합성마다
> 활성 패널 커서 셀의 전역 좌표로 `App.cursor_position` 을 설정해, Textual 이 프레임 끝마다
> `move_to(cursor_position)` 를 출력하도록 했다(Textual 의 Input/TextArea 가 IME 를
> 그렇게 처리하는 것과 동일 패턴). 모달(Input/TextArea)이 떠 있으면 그 위젯이
> `cursor_position` 을 소유하므로 코어는 건드리지 않는다. **좌표 동기화 로직은 단위
> 테스트로 가드**(`test_client.test_ime_hardware_cursor_follows_active_pane_cursor`)하나,
> preedit 오버레이 자체는 OS 그림이라 **실기(實機) 수동 확인이 필요**하다(아래 §6).
>
> _아래 §3~§5 의 원인 분석/수정 시나리오는 구현 근거로 그대로 보존한다._
> **증상 보고**: 한글 모드로 타이핑할 때, 글자는 커서 위치에 정상 표시되지만 **아래쪽
> 패널 아웃라인(테두리/구분선)의 같은 칸에 조합 중인 한글이 한 글자만** 뜨고, **마지막
> 입력 문자로 계속 바뀐다**. 타이핑 글자는 커서 위치에만 보여야 한다.
> **핵심 결론**: pytmux 는 활성 패널 커서를 **반전(reverse) 셀**로만 그리고 **실제
> 터미널(하드웨어) 커서를 활성 패널 위치로 옮기지 않는다**. 호스트 터미널(Windows
> Terminal 등)은 **IME 조합 문자열(preedit)을 하드웨어 커서 위치에 덧그리므로**, 커서가
> 엉뚱한 곳(테두리 행)에 박제돼 그 자리에 조합 글자가 라이브로 보인다. **수정 가능**하나
> (하드웨어 커서를 활성 패널 커서에 동기화), **헤드리스 테스트로는 검증 불가**(아래 §6).

---

## 1. 재현(Repro)

1. Windows Terminal(또는 IME preedit 을 커서 위치에 오버레이하는 터미널)에서 pytmux 실행.
2. Claude Code 가 도는 패널(또는 임의 셸 패널)에 포커스.
3. **한글 IME 를 켠 채** 입력창에 한글을 타이핑(예: "ㅁㄴㅇ…" 조합).
4. 관찰:
   - ✅ **커서 위치**: 입력/조합 글자가 정상적으로 보인다(내부 PTY 가 에코).
   - ❌ **패널 아웃라인(테두리/구분선)의 같은 칸**: 조합 중인 음절이 **한 글자**만 떠
     있고, 키를 칠 때마다 **마지막 글자로 계속 바뀐다**(스크린샷의 ` > ㅁ ㄴ ㅇ 2 …`
     라인과 그 위/아래 경계선에 박힌 잔상).

영문 입력에선 보이지 않거나 덜 두드러진다 — 영문은 키마다 즉시 확정(commit)돼 preedit
구간이 거의 없는 반면, 한글(두벌식)은 **음절을 조합하는 동안 preedit 이 살아 있어** 그
잔상이 또렷이 보인다.

## 2. 두 개의 "커서"가 따로 노는 구조

pytmux 화면에는 **시각적 커서**와 **하드웨어 커서**가 따로 있다.

- **시각적 커서(pytmux 가 그림)**: 합성기(`_composite`)가 모든 패널을 하나의 셀 그리드로
  합치고, 활성 패널 커서 셀을 **반전 스타일**로 칠한다 — `client.py:1300-1305`.
  ```python
  if cursor and p["id"] == active:
      ccx, ccy = cursor
      gx, gy = p["x"] + ccx, p["y"] + ccy
      if 0 <= gx < W and 0 <= gy < H:
          ch, st = cells[gy][gx]
          cells[gy][gx] = (ch, _with_reverse(st))   # 반전 = '가짜' 블록 커서
  ```
  이는 셀 색만 반전한 **그림**이다. 실제 터미널 커서를 옮기지 않는다.

- **하드웨어 커서(터미널이 소유)**: `MultiplexerView` 는 `render_line` 으로 셀만 출력하고
  (`clientwidgets.py:136`), **커서 위치를 보고하지 않는다**(`cursor_position`/`show_cursor`/
  preedit 처리가 코드 전역에 전혀 없음 — 확인됨). Textual 은 커서를 노출하지 않는 포커스
  위젯에 대해 하드웨어 커서를 **숨기거나 직전 기록 위치에 방치**한다.

내부 앱(Claude PTY)이 보내는 커서 이동·표시 시퀀스는 pytmux 가 **패널별 `cursor` 값으로
파싱해 보관**할 뿐(서버 `screen` 메시지 → `pane_content[(rows, cursor)]`,
`client.py:734`), **호스트 터미널로 forwarding 하지 않는다**. 따라서 실제 커서는 활성 패널
의 입력 지점으로 **결코 이동하지 않는다**.

## 3. 근본 원인 — IME preedit 은 하드웨어 커서에 그려진다

IME(입력기)의 **조합 문자열(preedit/마킹 텍스트)** 은 애플리케이션이 그리는 게 아니라
**호스트 터미널/OS 가 하드웨어 커서 위치에 오버레이**한다. pytmux 는 하드웨어 커서를 활성
패널 커서로 옮기지 않으므로:

1. 사용자가 한글을 조합하면 OS 가 preedit("ㅁ"→"마"→…)을 **하드웨어 커서가 박제된 자리**
   (Textual 이 직전에 마지막으로 쓴 위치 — 흔히 합성 그리드 경계/구분선 행)에 그린다.
2. 그 자리엔 **현재 조합 중인 음절 하나만** 뜨고, 키 입력마다 그 음절이 갱신되므로
   **"마지막 글자로 계속 바뀌는 한 글자"** 로 보인다.
3. 확정(commit)된 글자는 내부 PTY 가 에코해 **패널 커서 위치**에 정상적으로 찍힌다.

→ 결과적으로 **조합 글자(preedit)는 박제된 하드웨어 커서 자리(테두리)**, **확정 글자는
패널 커서 자리** 로 갈라져 보인다. 열(column)이 우연히 겹쳐 보이는 것은 Textual 이 마지막
출력 열 부근에 커서를 두기 때문이다.

증상의 3요소가 모두 설명된다: ① 한 글자만(=조합 중 음절), ② 마지막 입력으로 계속 변경
(=preedit 갱신), ③ 테두리/아웃라인 위치(=박제된 하드웨어 커서 행).

## 4. 수정 시나리오(제안) — 하드웨어 커서를 활성 패널 커서에 동기화

핵심 아이디어: **매 합성마다 실제 터미널 커서를 활성 패널의 시각적 커서 셀로 옮기고
보이게** 한다. 그러면 OS 가 preedit 을 **사용자가 타이핑하는 바로 그 자리**에 덧그려,
잔상이 사라지고 조합 글자가 커서 위치에 자연스럽게 합쳐진다.

후보 구현(우선순위 순):

- **(A) Textual 스크린 커서 API 사용(권장 1순위)**: 활성 패널 커서 셀 좌표를 Textual 이
  하드웨어 커서로 띄우도록 노출한다. `MultiplexerView` 가 커서를 보고하게 하거나
  (`Widget` 에 커서 좌표 제공), `App`/`Screen` 의 커서 위치를 매 `_composite` 마다
  `(gx, gy)` 로 설정. Textual 버전별 지원 API(예: 스크린 compositor 의 cursor, 또는
  focus 가능한 0크기 프록시 위젯을 커서 셀에 배치)를 확인해 채택.
- **(B) 드라이버에 ANSI 직접 출력(폴백)**: 프레임 출력 직후 `\x1b[{y};{x}H\x1b[?25h`
  (커서 이동+표시)를 드라이버로 써서 하드웨어 커서를 패널 커서 셀에 둔다. Textual 이 매
  프레임 커서를 숨기/이동할 수 있어 **순서·재정의에 취약** — 마지막 write hook 필요.
- **보강**: 하드웨어 커서가 패널 커서와 일치하면, 기존 **반전 블록 커서는 중복**이 된다.
  (a) 반전 커서를 끄고 하드웨어 커서만 쓰거나, (b) 두 표시가 겹쳐도 무방하면 유지.
  비활성/스크롤/팝업 상태에선 하드웨어 커서를 **숨겨**(`\x1b[?25l`) preedit 이 엉뚱한 데
  안 뜨게 한다.

부수 효과(긍정): 하드웨어 커서가 올바른 위치로 가면, IME 후보창(candidate window) 위치도
정확해지고, 스크린리더/접근성 커서 추적도 개선된다.

### 경계 조건(설계 시 반영)
- 활성 패널이 **대체화면(alt-screen)·스크롤백 보기·모달 팝업** 중일 때의 커서 표시 정책.
- 패널 커서가 **헤더 예약 행 보정**(`claude_hdr` 로 내용이 한 행 내려감, `client.py:1034`)
  과 어긋나지 않게 좌표 계산.
- **와이드 문자(한글) 셀**: 커서가 2칸 글자 위면 시작 칸 기준으로 좌표 설정.
- degraded/리페인트 합치기 등 부분 갱신 경로에서도 매 프레임 커서 좌표가 최신이어야 함.

## 6. 검증의 한계(중요) — 왜 "시나리오"인가
IME preedit 은 **셀 텍스트 버퍼에 들어오지 않는 OS/터미널 오버레이**다. pytmux 의 헤드리스
하네스는 **텍스트 화면만 캡처**(text screenshot)하고 **OS 픽셀 캡처는 차단**돼 있어
(메모리: run-pytmux 드라이버 노트), **이 잔상은 자동 테스트로 재현·검증할 수 없다**. 즉
코드 수정만으로는 "고쳐졌다"를 하네스로 증명할 수 없고 **실제 Windows Terminal + 한글
IME 수동 확인**이 필요하다. 그래서 본 작업은 (성급한 미검증 코드 변경 대신) **원인 규명과
수정 시나리오 문서화**로 제출한다. 회귀 보조로는 "활성 패널 커서 셀 좌표(gx,gy) 산출"
로직에 단위 테스트를 둘 수 있으나(하드웨어 커서 좌표가 패널 커서와 일치하는지), preedit
오버레이 자체는 수동 검증 대상이다.

---

## 7. 한 줄 요약
pytmux 가 **하드웨어 커서를 활성 패널 커서로 옮기지 않아**(반전 셀로만 그림), 호스트
터미널이 **IME 조합 글자를 박제된 커서 자리(테두리)에 덧그린다**. 수정안은 **매 합성마다
하드웨어 커서를 활성 패널 커서 셀에 동기화**하는 것이며, preedit 오버레이 특성상 **검증은
실기(實機) 수동 확인**이 필요하다.
