# 세션에서 배운 점 — 2026-06-09 (ncd/NCD, 999% 버그, captures 머신명 이관, 데몬 재시작)

> 이 문서는 "무엇을 만들었나"(그건 NC_SCENARIO·HANDOFF 에 있음)가 아니라, **다음에
> 시간을 아껴 줄 비자명한 운영·구현 교훈**을 모은다. 구체 명령·파일·라인으로 남긴다.

## 1. Perforce + git 듀얼 퍼블리시 — 실전 절차

- 이 저장소는 **p4 가 1차**(권위), git 은 미러다. 둘은 **양방향으로 lag** 한다
  ([[publish-workflow-p4-and-git]]). 한쪽만 보고 "최신"이라 단정하지 말 것.
- **의미 있는 단계마다 별도 번호 CL** 로 서브밋(디폴트 체인지리스트 금지). 빈 번호
  CL 을 먼저 만들고 파일을 그쪽으로 연다:
  ```bash
  NEW=$(printf "Change: new\nDescription:\n\t<desc>\n" | p4 change -i | grep -oE '[0-9]+')
  p4 edit -c "$NEW" <files>   # 기존 파일
  p4 add  -c "$NEW" <files>   # 새 파일
  p4 submit -c "$NEW"
  ```
- **함정**: 이미 파일이 열린 번호 CL 의 설명만 고치려고 `p4 change -i` 에 `Change: <n>`
  스펙을 넣되 **Files: 섹션을 빠뜨리면 그 파일들이 default 로 빠진다**. 그러면 submit 이
  "No files to submit". 복구: `p4 reopen -c <n> <files>` 후 재submit. 설명은 처음
  `p4 change -i` 만들 때 다 넣는 게 안전.
- **git 미러 시 WIP 보존**: 작업 트리에 미커밋 변경(다른 사람/다른 CL 의 WIP)이 있으면
  `git pull --rebase` 가 막힌다. 내 파일만 commit 한 뒤 나머지는 stash:
  ```bash
  git add <my-files>; git commit -m ...
  git stash push -q -- pytmuxlib/ tests/   # WIP 만 치움
  git pull --rebase -q && git push
  git stash pop
  ```
- **captures/ 는 git 미러 제외**(.gitignore). captures 관련 작업은 **p4 전용 서브밋**
  (git push 하지 말 것). 반대로 코드/문서는 p4+git 양쪽.

## 2. 체인지리스트 격리 — 남의 WIP 와 안 섞기

- 세션 시작 시 다른 작업의 WIP 가 번호 CL 로 열려 있을 수 있다(이번엔 CL 57699 =
  claude.py·clientscreens.py·model.py·serverclaude.py·test_claude.py·test_server.py).
- **p4 는 한 파일을 한 CL 에만 둘 수 있다.** 그 WIP 파일을 내 변경에도 손대야 하면
  내 변경이 그 WIP 와 한 묶음이 된다(분리 불가). → **회피책**:
  - 새 화면/모듈은 **별 파일**로 만든다. 이번에 ncd 모달을 `clientscreens.py`(57699 점유)
    에 넣지 않고 **`pytmuxlib/clientnc.py`** 신설 → CL 격리 성공.
  - 소스 파일이 WIP 로 잠겼는데 거기에 한 줄 고쳐야 하면(예: 999% 의 `_tok5h_pct` 가
    `serverclaude.py`=57699 안), **깨끗한 다른 지점**에서 고친다. 999% 는 서버 산출부
    대신 **클라 표시 경계 `clientwidgets.update_status`** 에서 클램프해 격리했다.
- 작업 후 `p4 opened` 로 **default 가 비었고, 열린 건 원래 WIP 뿐**인지 확인.

## 3. `p4 sync` 후 resolve, 그리고 라이브 데몬 주의

- sync 시 내가 연 파일이 depot 최신보다 뒤면 `must resolve` 가 뜬다. 충돌 영역이
  다르면 **자동 병합**으로 끝난다: `p4 sync <file>; p4 resolve -am <file>`. 병합 후
  **내 추가분이 살아남았는지 grep 으로 확인**하고 테스트 재실행.
- `p4 sync` 는 **opened 파일(WIP)은 안 건드린다**(resolve 만 예약). 그래서 WIP 는
  보존되지만, 그 WIP CL 을 나중에 submit 하려면 resolve 가 필요해진다 — 남의 WIP 면
  **resolve 하지 말고 그대로 두고** 보고만 한다.
- 전체 sync 는 **다른 머신의 코드**(office CL 등)를 디스크로 끌어와 git 작업트리를
  어지럽힌다(git HEAD 엔 없는 변경이 modified 로 보임). 이때 git 미러는 조심.

## 4. captures 머신명 체계 (default → `<hostname>`)

- 경로 규칙: `servercapture.capture_dir` 는 **기본 소켓일 때만**
  `captures/<머신이름>/`(`_capture_subdir` = `_safe(socket.gethostname())`), 비기본
  소켓은 `<state>.capture`. 이 머신 hostname=`playground.local`, office 머신=`woojinkim`.
- **여러 머신이 같은 captures/ 를 p4 로 공유**하므로 머신별 디렉터리로 격리한다.
  기존 `captures/default` 로그를 머신명 하위로 옮기는 **머신별 이관**이 필요(CL 57744 가
  office=woojinkim, 본 세션이 playground.local).
- **이관은 rename 으로**: 데몬이 캡처 파일 fd 를 캐싱하므로(`_close_capfile`),
  `mv captures/default/*.log captures/<host>/` 하면 **열린 fd 가 inode 를 따라가** 라이브
  로그가 끊기지 않고 새 위치에 계속 append 된다(검증: 옮기는 중 파일 크기 증가 지속).
- depot 에서 이미 삭제된 orphan(예: pane-1~5·sessions.log, CL 57744)은 **되살리지 말 것**.
  머신명 디렉터리엔 현행/타임스탬프 로그만.
- captures 는 git 무시지만 **p4 add 는 됨**(중앙 p4ignore 가 captures 를 막지 않음;
  [[p4ignore-single-central-file]]).

## 5. 작업 보존 서버 재시작(re-exec) 운영

- **드라이런 우선이 강제된다**: auto-mode 분류기가 라이브 데몬 재시작을 막고
  "restart-check 를 먼저 돌리고 사용자 확인" 을 요구한다([[restart-dryrun-first-protocol]]).
  거스르지 말고 절차를 따른다.
- **restart-check 는 제어 CLI(`pytmux.py cmd`)에 없다**(handle_control 미등록). 대신
  **`/tmp/restart_dryrun.py`** 가 `request_restart_check` 를 라이브 기본 서버에 그대로
  보내 PASS/FAIL 팝업을 텍스트로 재현한다(부작용 없음). 이걸로 SAFE 확인 → 사용자
  승인 → `python3 pytmux.py cmd restart-server`.
- **re-exec 는 PID 를 유지**한다(`ps` 의 lstart 는 최초 기동 시각 그대로). "재시작
  안 된 듯" 보여도 **버전으로 확인**: 드라이런 헤더의 `실행=p4:NNNNN  디스크=p4:MMMMM`
  이 재시작 후 **동일**해지면 새 코드 적용 완료.
- **캡처 파일 fd 는 execv 때 닫힌다**(master PTY fd 만 CLOEXEC 해제로 보존). 그래서
  재시작 직후 새 코드가 **새 capture_dir 로 새 파일을 연다**. 전환 순간에 옛 경로
  (default)에 **작은 frozen 잔재**(수십 바이트)가 남을 수 있다 — 새 경로의 완전한
  파일이 있으니 그 잔재는 지워도 된다(미추적이면 그냥 `rm`; `p4 fstat`/`p4 have` 로
  headAction delete·not on client 확인 후).
- 검증법: 재시작 전 `touch /tmp/marker` → 활동 유발 → `find <dir> -newer /tmp/marker`
  로 **어느 디렉터리가 새로 기록되는지** 본다. 살아 커지는 파일(mtime 갱신·size 증가)이
  현재 capture_dir.

## 6. Textual 함정 (clientnc 작업에서)

- **메서드 이름이 Textual 내부와 충돌하면 렌더가 깨진다.** 위젯 메서드를 `_render`
  로 지었더니 Textual 이 위젯 시각화로 그 코루틴을 호출해
  `'coroutine' object has no attribute 'render_strips'` 크래시. → `_rebuild` 로 개명.
  위젯 서브클래스에서 `render`/`_render`/`render_line` 등 **프레임워크 예약명 회피**.
- **ListView 하이라이트 클래스는 `.-highlight`**(대시 1개), 포커스 시
  `&:focus > ListItem.-highlight`. 색 입힐 때 이 선택자 사용.
- **ListView 는 이미 비애니메이션 스크롤**(`scroll_to_widget(..., animate=False)`).
  ssh 스크롤이 느린 건 애니메이션이 아니라 **스크롤 시 뷰포트 전체 리페인트** 때문.
- **빠른 ssh 트리 = `render_line` 단일 위젯**(MultiplexerView 방식). 항목마다 위젯
  (ListItem) 두는 ListView 대신, 커서·스크롤을 직접 관리하고 **↑↓ 시 바뀐 두 줄만
  `refresh(Region(...))`**, 스크롤될 때만 전체 refresh. 위젯 레이아웃 비용 0.
  와이드(한글) 문자는 `Strip(...).adjust_cell_length(width, style)` 로 셀 패딩.
- **헤드리스 테스트**: `app.run_test()` 로 pilot, `app._dispatch({"t":...})` 로 서버
  메시지 주입, `app.send_cmd`/`send_input` 람다 모킹, `pilot.press("키")` 또는
  위젯 `on_key(Key("shift+enter", None))` 직접 호출(터미널이 못 주는 조합 검증).

## 7. 잡다한 확인 사항

- **테스트는 pytest 아님**: `python3 tests/run.py [모듈...]`(헤드리스 async 러너).
  test_*.py 의 async `test_` 함수를 각자 루프에서 실행. ([[run-pytmux]] 스킬도 참고)
- **Windows cd 의 드라이브 전환**: `cd D:\x` 는 cmd.exe 에서 드라이브를 안 바꾼다 →
  **`cd /d "D:\x"`**. 기본 셸은 COMSPEC=cmd.exe(`serverpty._fork_shell`). OS별
  분기는 `client._ncd_cd_command`. (새 패널 분할은 새 셸을 그 cwd 로 띄워 셸 무관 전환.)
- **모호한 한 줄 지시는 코드부터 확인**: "999%" 는 `min(999, …)` 클램프가 그대로
  노출된 것(분모<<사용량). 신고값(999·드라이브·머신명)은 **원인 라인을 grep 으로
  특정**한 뒤 고친다([[claude-scrape-false-positives]] 의 "원시바이트부터" 와 같은 결).
