# pytmux — Python/Textual 기반 tmux 유사 터미널 멀티플렉서 설계

> 상태: Draft v1
> 작성일: 2026-06-02
> 대상: `scripts/pytmux`

## 1. 개요

`pytmux` 는 Python 과 [Textual](https://textual.textualize.io/) 로 구현하는 터미널 멀티플렉서다.
tmux 와 동일하게 하나의 터미널 안에서 여러 셸 세션·패널을 동시에 띄우고, 분할/이동/리사이즈하며,
**앱이나 상위 터미널을 닫아도 셸 세션이 그대로 살아있게** 한다.
기존 tmux 와의 가장 큰 차별점은 **마우스 1급 지원**(드래그로 리사이즈, 클릭으로 패널 추가/삭제)과
**TUI 메뉴 기반 제어**를 전면에 둔다는 점이다.

이 문서는 구현 전 설계 합의를 위한 것이며, 코드는 포함하지 않는다.

## 2. 요구사항

### 2.1 기능 요구사항 (사용자 요청 기준)

| # | 요구사항 | 충족 방식 요약 |
|---|----------|----------------|
| R1 | 앱을 시작하면 일반 터미널과 동일하게 보여야 한다 | 시작 시 단일 패널(전체화면)에 사용자의 기본 셸을 그대로 띄움 |
| R2 | 앱을 닫거나 상위 터미널을 닫아도 세션이 유지되어야 한다 | 셸 PTY 를 보유하는 **백그라운드 데몬(서버)** 과 화면을 그리는 **클라이언트** 를 분리 |
| R3 | 화면 하단에 tmux 와 동일한 상태 표시줄 1줄 | 클라이언트 하단에 `StatusBar` 위젯(세션명·탭 목록·시각 등) |
| R4 | 새 세션 열기, 패널 추가/삭제/이동/리사이즈를 모두 TUI 메뉴로 제어 | prefix 키 + 명령 팔레트 + 팝업 메뉴 |
| R5 | 마우스로 리사이즈·삭제·추가 가능 | 분할선 드래그 리사이즈, 패널 헤더의 닫기 버튼, 분할 버튼/우클릭 컨텍스트 메뉴 |
| R6 | 패널별로 위로 스크롤하여 화면 밖으로 밀려난 이전 내용을 볼 수 있어야 한다 | 패널마다 스크롤백 버퍼 보유. 휠/키로 위로 스크롤 시 지난 출력 표시 |

### 2.2 비기능 요구사항

- **이식성**: macOS / Linux (POSIX PTY). Windows 는 1차 범위 외(ConPTY 는 추후 과제).
- **단일 의존 최소화**: 표준 라이브러리 PTY + `pyte`(터미널 에뮬), `textual`(TUI). 외부 데몬 불필요.
- **응답성**: 키 입력 → 화면 반영 지연을 체감 불가 수준(<16ms 목표 프레임)으로.
- **복구력**: 클라이언트가 죽어도 서버는 무손상. 서버가 죽으면 세션은 사라지되(영속 셸 프로세스는 OS 가 정리) 안전하게 종료.

## 3. 핵심 설계 결정 — 왜 클라이언트/서버로 나누는가

R2(세션 영속성)가 전체 아키텍처를 결정한다.

Textual 앱은 그 자체가 "현재 터미널"에 붙어 동작하는 **프론트엔드**다. 사용자가 터미널 창을 닫으면
Textual 프로세스는 SIGHUP 등을 받고 종료된다. 따라서 셸 세션(자식 프로세스 + PTY)이 Textual 앱과
**같은 프로세스/프로세스 그룹**에 묶여 있으면 같이 죽는다.

해결책은 tmux 와 동일하다: 셸을 들고 있는 부분을 **별도의 데몬 프로세스**로 떼어내 컨트롤링 터미널에서
분리(detach)한다.

```
요청한 "앱을 닫아도 세션 유지" =  셸 PTY 의 수명을 클라이언트 수명에서 분리한다
                              =  서버(데몬) 가 PTY 를 소유한다
```

- **서버(데몬)**: 세션/윈도우/패널 트리와 각 패널의 PTY·자식 셸·화면 버퍼를 소유. 컨트롤링 터미널과
  분리되어 백그라운드에서 계속 실행. 유닉스 도메인 소켓으로 명령/화면을 주고받는다.
- **클라이언트(Textual)**: 소켓에 attach 하여 화면을 그리고, 키/마우스를 명령으로 변환해 서버에 보낸다.
  종료(detach)해도 서버와 셸은 그대로 남는다. 다시 실행하면 같은 소켓에 attach 하여 이어서 본다.

이 구조 덕분에:
- 클라이언트 0개여도 셸은 계속 돈다(빌드/SSH 세션 유지).
- 여러 클라이언트가 같은 세션에 동시 attach 가능(미러링).

│   ┌──────────────────── 클라이언트 (Textual App, 포그라운드) ────────────────────────┐   │

## 5. 컴포넌트 상세

### 5.1 런처 / 부트스트랩 (`pytmux` 진입점)

1. 소켓 경로 결정: `${XDG_RUNTIME_DIR:-/tmp}/pytmux-$UID/<server>.sock`.
2. 소켓에 connect 시도.
   - 성공 → 기존 서버 발견. 클라이언트로 attach.
   - 실패(없음) → 서버를 **데몬으로 fork** 후, 소켓이 뜰 때까지 대기, attach.
3. 데몬화 절차(서버): `fork → setsid → fork`(이중 포크)로 세션 리더 분리, `chdir`, stdio 를 `/dev/null`
   로 리다이렉트, 컨트롤링 터미널 분리. 이후 서버 이벤트 루프 진입.

부가 서브커맨드: `pytmux attach`(단일 세션), `pytmux ls`(탭/패널 요약),
`pytmux kill-server`, `pytmux cmd <명령>`(외부 제어), `pytmux record|replay`(렌더 진단).

### 5.2 서버: 세션 모델

데이터 트리(메모리 상주):

```
Server
 └─ sessions: dict[name → Session]
Session
 ├─ name, created_at
 └─ tabs: list[Tab], active_tab  (active_window = active_tab.window 호환 프로퍼티)
Tab                                  # 최상위 전환 단위(상태표시줄의 탭)
 ├─ index, name
 ├─ has_activity/has_bell, monitor_activity/monitor_bell
 └─ window: Window                   # 탭에 종속된 단일 윈도우(1:1)
Window                               # 패널 집합을 보유하는 렌더 영역
 ├─ zoomed, sync, border_status, auto_rename
 └─ root: PaneNode (이진 분할 트리의 루트)
PaneNode (둘 중 하나)
 ├─ Split{ direction: H|V, ratio: float, child_a, child_b }   # 내부 노드
 └─ Pane{ id, pty_fd, child_pid, screen(pyte), cols, rows, title, cwd }  # 잎 노드
```

- **Tab(최상위)**: 전환 단위. 이름/인덱스(상태표시줄 탭)와 활동/벨 표시를 보유하고,
  정확히 하나의 Window 를 종속으로 가진다. `new-tab`(=`new-window`)으로 새 탭을 만들면
  새 윈도우(단일 패널)가 생기고 이를 패널로 분할한다. `kill-tab`/`rename-tab` 명령 제공.
- **Window(중간)**: 패널 분할 트리와 줌/동기화/모니터 상태를 보유. `split-window` 가 이
  윈도우를 패널로 나눈다.
- **Pane(잎)**: 실제 셸 1개 = PTY master fd + 자식 PID + `pyte.Screen` 버퍼.
- **Split(내부)**: 수평/수직 분할과 분할 비율(`ratio`, 0~1)을 보유. 이진 트리로 임의 중첩 분할 표현.
- 패널 식별자는 tmux 처럼 `%0, %1, ...` 단조 증가 ID.

### 5.3 PTY 와 셸 프로세스

- 패널 생성 시 `pty.openpty()` 로 master/slave 쌍 생성.
- `fork` 후 자식에서 `setsid()` → slave 를 컨트롤링 터미널로 지정 → `dup2`(stdin/out/err) → `execvp(셸)`.
- 환경변수: `TERM=xterm-256color`, `PYTMUX=<socket>`, 사용자 환경 상속.
- master fd 는 non-blocking 으로 asyncio 루프에 등록(`loop.add_reader`).
- 창 크기 변경: 레이아웃 계산으로 패널 (cols, rows) 산출 → `fcntl.ioctl(master, TIOCSWINSZ, ...)` →
  자식에 SIGWINCH 자동 전달 → 동시에 `pyte.Screen.resize()`.

### 5.4 터미널 에뮬레이션 (`pyte`)

서버는 셸의 원시 출력 바이트를 그대로 클라이언트에 흘리지 않는다. 대신 패널마다 `pyte` 가상 스크린을
두고, master 에서 읽은 바이트를 `pyte.Stream`/`ByteStream` 에 feed 한다. **스크롤백(R6)을 위해 패널마다
`pyte.HistoryScreen` 을 사용**하여, 위로 밀려난 행을 history 버퍼에 누적 보관한다. `HistoryScreen` 은
보이는 셀 격자(문자 + 색/속성) + 커서 + 화면 위쪽으로 스크롤된 과거 행들을 함께 유지한다.

이렇게 하는 이유:
- 패널은 화면의 일부 직사각형일 뿐이므로, **현재 스크린 상태(격자)** 를 알아야 그 영역만 다시 그릴 수 있다.
- 클라이언트가 재attach 하면 누적 스트림이 아니라 **현재 화면 전체 스냅샷**만 받아 즉시 복원 가능.
- diff 전송(변경된 행/셀만)으로 대역폭·렌더 비용 절감.

한계/주의: pyte 가 모든 제어 시퀀스를 100% 지원하지는 않는다(일부 그래픽/사설 모드). 1차 범위는 일반
셸·vim·less·htop 수준 호환을 목표로 하고, 미지원 시퀀스는 무시 후 추후 보강.

**대체 화면 버퍼(alternate screen)**: pyte 는 `ESC[?1049h/l`(및 1047/47)을 지원하지 않는다.
vim·less·htop·Claude Code 등 풀스크린 TUI 는 진입 시 대체 화면으로 전환하므로, 미지원이면
TUI 의 그리기가 기존 셸 스크롤백 위에 겹쳐 화면이 깨진다. 따라서 pytmux 가 `Pane` 레벨에서
직접 처리한다: 메인 화면(`HistoryScreen`, 스크롤백)과 대체 화면(`Screen`, 스크롤백 없음)을 두고,
PTY 출력 스트림에서 전환 시퀀스를 가로채 활성 화면을 바꾼다(`feed` 가 마커로 분할 라우팅,
경계에서 잘린 시퀀스는 다음 read 로 이월). 대체 화면을 떠나면 메인 화면(셸 내용)이 그대로 복원된다.

### 5.5 IPC 프로토콜

- **전송**: 유닉스 도메인 소켓, 길이-프리픽스(4바이트 빅엔디언) + JSON 또는 MessagePack 페이로드.
  화면 셀 데이터는 양이 많으므로 MessagePack 권장(2차 최적화).
- **클라이언트 → 서버 (명령/입력)**:
  - `attach{session}` / `detach`
  - `input{pane_id, data(bytes)}` — 키 입력 원시 바이트
  - `resize_client{cols, rows}` — 클라이언트(전체 터미널) 크기 변경
  - `scroll{pane_id, delta | to_top | to_bottom}` — 패널 스크롤백 뷰포트 이동(R6, §5.10)
  - `cmd{...}` — 제어 명령(아래 §5.8): split / kill-pane / select-pane / resize-pane / new-window(=새 탭) / rename / layout 저장·불러오기 ...
- **서버 → 클라이언트 (이벤트/렌더)**:
  - `layout{window 트리 + 각 패널의 사각형 좌표(x,y,w,h)}`
  - `screen_update{pane_id, 변경 행/셀, 커서}` (전체 또는 diff)
  - `status{세션·윈도우·패널 메타}` — 상태표시줄 갱신용
  - `bell{pane_id}`, `pane_exit{pane_id, status}`
- **프레임 동기화**: 서버는 PTY 읽기 후 dirty 패널을 모았다가 ~60Hz 로 묶어서 push(coalescing).

### 5.6 클라이언트: Textual 앱 구성

```
PytmuxApp(App)
 ├─ Screen
 │   ├─ WindowContainer            # 패널 레이아웃 영역 (전체 - 하단 1줄)
 │   │   ├─ PaneView(%0)           # 커스텀 위젯: pyte 스냅샷을 Rich 라인으로 렌더
 │   │   ├─ Divider(...)           # 분할선(드래그 가능, 마우스 타깃)
 │   │   └─ PaneView(%1)
 │   └─ StatusBar                  # 하단 1줄
 └─ CommandPalette / PopupMenu     # 메뉴 기반 제어
```

- **PaneView**: 서버에서 받은 셀 격자를 매 프레임 `Rich` 텍스트 라인으로 변환해 그린다. 색/속성 매핑.
  포커스된 패널은 테두리/헤더 강조. 헤더(선택)에는 패널 제목과 닫기(✕) 버튼.
- **레이아웃 배치**: 서버가 보낸 `layout`(각 패널의 x,y,w,h)을 그대로 절대 배치. Textual 의
  컨테이너 대신 좌표 기반 배치로 서버-클라이언트 레이아웃 단일 소스 보장(서버가 권위자).
- **입력 처리**: 포커스 패널로 가는 키는 raw 바이트로 인코딩해 `input` 으로 전송. 단, **prefix 키**(기본
  `Ctrl-b`)는 가로채 메뉴/명령 모드로 진입.

### 5.7 상태 표시줄 (R3)

하단 고정 1줄(`StatusBar` 위젯). tmux 기본 스타일을 모사:

```
[session-name] 0:zsh* 1:vim  2:logs                                  "host"  14:32 02-Jun-26
└ 좌: 세션명     └ 윈도우 목록(활성=*)                                 └ 우: 호스트 / 시계
```

- 중앙: 탭 인덱스:이름 목록(활성 강조). 우측: 활성 패널 제목·호스트명·시각.
  단, **상단 탭바가 보이면 하단의 탭 목록은 생략**한다(중복 방지).
- 상단 탭바(별도 위젯): 탭이 2개↑이거나 `tab-bar always` 설정 시 표시. 탭 목록 +
  마지막 탭 오른쪽 [+] 버튼, 폭 초과 시 ◀▶ 스크롤. 탭 닫기 [x] 는 콘텐츠 영역
  오른쪽 위 모서리로 분리. 마우스 클릭·ESC 모드 방향키로 조작.
- 데이터는 서버 `status` 이벤트로 갱신. 1초 주기 시계 틱은 클라이언트 로컬.
- 색상은 p4v-tui 와 동일한 Textual textual-dark 테마 팔레트를 따르며 설정으로 덮어쓰기 가능.

### 5.8 메뉴/명령 시스템 (R4)

세 가지 진입 경로를 제공한다.

1. **Prefix 키**: `Ctrl-b` 입력 시 명령 모드. 이후 단일 키가 tmux 와 동일하게 매핑:
   - `"` 수평 분할, `%` 수직 분할, `x` 패널 삭제, `o`/방향키 패널 이동,
     `c` 새 윈도우, `n`/`p` 윈도우 전환, `d` detach, `,` 이름변경,
     `Ctrl-방향키` 리사이즈.
2. **명령 팔레트**: Textual `CommandPalette`(`Ctrl-b :` 또는 단축키)로 명령명을 타이핑/검색해 실행.
   모든 동작이 named command 로 등록되어 팔레트·키바인딩·마우스가 동일한 명령을 호출.
3. **팝업 메뉴**: prefix 후 `Enter` 또는 마우스 우클릭으로 컨텍스트 메뉴(분할/삭제/이동/리사이즈/
   새 세션/이름변경 항목 리스트)를 띄워 키보드·마우스 모두로 선택.

명령은 클라이언트에서 `cmd{action, args}` 로 서버에 전달되고, 서버가 트리를 수정한 뒤 새 `layout` 을 push.

### 5.9 마우스 인터랙션 (R5)

Textual 의 마우스 이벤트(`MouseDown/Move/Up`, `Click`, `Enter/Leave`)를 사용:

- **리사이즈**: `Divider` 위젯 위에서 드래그 → 드래그 변위를 비율 변화로 환산해 `resize-pane` 명령 전송.
  드래그 중 커서를 `↔`/`↕` 로 변경, 실시간 미리보기.
- **포커스 전환**: 패널 본문 클릭 → 해당 패널 `select-pane`.
- **삭제**: 패널 헤더의 `✕` 버튼 클릭 → 확인 후 `kill-pane`.
- **추가**: 패널 헤더의 분할 버튼(⬍/⬌) 또는 우클릭 컨텍스트 메뉴 → `split-window`.
- **스크롤백(R6)**: 포커스·비포커스 무관하게 **마우스 커서가 올라간 패널** 위에서 휠을 위로 굴리면
  그 패널만 스크롤백 모드로 전환되어 지난 출력을 보여준다. 휠 아래로 굴려 맨 아래(live)에 도달하면
  스크롤백 모드 자동 해제. 패널별 독립 스크롤(한 패널을 스크롤해도 다른 패널은 영향 없음).

마우스 좌표 → 패널/분할선 매핑은 서버가 보낸 layout 사각형으로 클라이언트가 히트테스트.

### 5.10 패널별 스크롤백 (R6)

tmux 는 위로 스크롤해 지난 화면을 보려면 별도의 copy-mode 로 진입해야 해서 직관성이 떨어진다.
`pytmux` 는 패널마다 **항상 활성화된 독립 스크롤백**을 제공해 휠/키만으로 곧바로 과거 출력을 본다.

- **버퍼 소유**: 서버의 각 `Pane` 이 `pyte.HistoryScreen` 으로 화면 밖으로 밀려난 행을 누적 보관.
  보관 한도(기본 예: 10,000 행, 설정 가능)를 두고 초과 시 가장 오래된 행부터 폐기.
- **뷰포트 상태**: 각 패널은 `scroll_offset`(0 = live 맨 아래, 양수 = 위로 N 행 거슬러 올라감)을 가진다.
  서버가 권위자이며 클라이언트는 이 값을 명령으로 변경 요청한다.
- **상호작용**:
  - 마우스: 패널 위에서 휠 위/아래.
  - 키보드: prefix 후 copy/scroll 모드(예: `Ctrl-b [`) 진입 → `PageUp/PageDown`, 방향키, `g`/`G`(맨 위/맨 아래).
  - 어느 경로든 `scroll{pane_id, delta | to_top | to_bottom}` 명령으로 통일.
- **렌더**: 스크롤백 모드에서는 서버가 해당 오프셋 구간의 행을 잘라 `screen_update` 로 보낸다.
  PaneView 는 우측에 스크롤 위치 표시(스크롤바/`[12/10000]` 등)와 "SCROLL" 표식을 띄운다.
- **live 출력 정책**: 스크롤백 보는 동안 셸이 새 출력을 내면 history 에는 계속 쌓되 **뷰포트는 고정**(자동
  점프 금지). 사용자가 맨 아래(live)로 복귀하면 그간 쌓인 내용을 한 번에 반영. (옵션: 맨 아래에 있을 때만
  자동 추종.)
- **입력 차단**: 스크롤백 모드에서는 키 입력을 셸로 전달하지 않는다(스크롤 조작 전용). 새 입력 시작/맨 아래
  복귀로 모드 해제.
- **리사이즈 상호작용**: 패널 리사이즈로 열 폭이 바뀌면 reflow 가 필요. 1차에서는 단순 잘림 허용,
  reflow 정확도 향상은 후속 과제.

## 6. 레이아웃 알고리즘

- 윈도우는 이진 분할 트리. 루트에서 시작해 가용 사각형을 자식 노드에 분배.
- 각 `Split` 은 방향(H/V)과 `ratio` 로 두 자식 사각형을 나눈다. 분할선/테두리 폭(1셀)을 고려해 정수 셀로 반올림.
- **신규 분할**: 대상 잎 Pane 을 `Split{ child_a=기존, child_b=새 Pane }` 로 치환.
- **삭제**: 잎을 제거하고 부모 Split 을 형제 노드로 대체(트리 수축). 마지막 패널 삭제 시 윈도우 닫힘,
  마지막 윈도우 닫히면 세션 종료.
- **리사이즈**: 해당 Split 의 `ratio` 변경 후 전 트리 재배치. 최소 크기(예: 3행×10열) 보장.
- 재배치 결과로 바뀐 모든 패널에 대해 PTY winsize + pyte resize 적용.

## 7. 영속성 (R2) 동작 시나리오

| 상황 | 결과 |
|------|------|
| 클라이언트에서 `detach`(Ctrl-b d) | 클라이언트 종료, 서버·셸 유지. 재실행 시 이어서 attach |
| 사용자가 터미널 창을 닫음(SIGHUP) | 클라이언트만 종료. 데몬은 컨트롤링 터미널과 분리되어 영향 없음 |
| 셸에서 `exit` | 해당 패널 종료 → 트리 수축. 모든 패널 종료 시 세션 정리 |
| 서버 명시적 종료(`kill-server`) | 모든 세션·셸 종료 |
| OS 재부팅 | 세션 소멸(메모리 상주). 영속화는 비목표(추후 레이아웃 직렬화로 부분 복원 검토) |

데몬이 컨트롤링 터미널에서 분리되는 것(`setsid` + 이중 포크 + stdio /dev/null)이 R2 의 기술적 핵심이다.

## 8. 의존성

- Python ≥ 3.11 (asyncio, 표준 `pty`/`fcntl`/`termios`/`select`)
- `textual` (TUI), `rich`(Textual 내장)
- `pyte` (터미널 에뮬레이션)
- (선택) `msgpack` (IPC 직렬화 최적화)

## 9. 디렉토리 구조 (현재)

진입점 `pytmux.py` 는 얇은 re-export 셸이고, 구현은 `pytmuxlib/` 패키지로 분리되어
있다(유지보수·LLM 친화성을 위해 역할별 모듈로 나눔).

```
scripts/pytmux/
├── pytmux.py                # 진입점(런처) — pytmuxlib 의 main 및 공개 API re-export
├── pytmuxlib/
│   ├── __init__.py
│   ├── protocol.py          # 상수·소켓 경로·프레이밍(read/write_msg)·색/리밋시각 헬퍼
│   ├── keymap.py            # 설정 파일 로드(load_config)·키 표기 변환
│   ├── model.py             # Pane/Split/Window/Session 트리 + 레이아웃·프리셋
│   ├── server.py            # Server(데몬): PTY·pyte·세션·명령·flush·영속
│   ├── client.py            # Textual 클라이언트(build_client_app): 렌더·키·마우스·메뉴
│   ├── replay.py            # 렌더 진단: record(PTY 녹화)·replay(텍스트 프레임 재생)
│   └── launcher.py          # 데몬화·attach·ls·kill-server·cmd·record·replay·main
├── tests/                   # 헤드리스 테스트(run.py) + 리플레이 골든 스냅샷
├── docs/                    # DESIGN / FEATURES / CONTRIBUTING / INPUT_FOCUS_NOTE
├── README.md
├── requirements.txt
└── pytmux.conf.example
```

## 10. 구현 마일스톤

1. **M1 — PTY 에코**: 단일 PTY 셸을 띄워 raw 입출력 왕복(서버 없이 단일 프로세스 PoC).
2. **M2 — 데몬 분리**: 서버 데몬화 + 소켓 + 클라이언트 attach/detach. R1·R2 충족(단일 패널).
3. **M3 — pyte 렌더 + 스크롤백**: PaneView 가 pyte 스냅샷을 정확히 렌더(vim/htop 검증) + `HistoryScreen`
   기반 패널별 스크롤백(R6) 휠/키 스크롤·뷰포트 고정.
4. **M4 — 분할/트리**: 수평·수직 분할, 패널 이동/삭제, 레이아웃 push. 상태표시줄(R3).
5. **M5 — 메뉴/키맵**: prefix 키 + 명령 팔레트 + 팝업 메뉴(R4).
6. **M6 — 마우스**: 드래그 리사이즈·클릭 포커스·버튼 삭제/추가(R5).
7. **M7 — 탭 + 다듬기**: 단일 세션 + 탭(상단 탭바), 탭별 레이아웃 슬롯, 복수 클라이언트 미러링.
   (멀티 세션은 사용자 표면에서 제거 — §5.2 참조.)

## 11. 위험 요소 및 대안

- **터미널 에뮬레이션 정확도**: pyte 의 시퀀스 커버리지 한계 → 호환성 테스트 스위트(vim/less/htop/tmux 자체)
  로 점검. 부족 시 자체 시퀀스 핸들러 보강.
- **렌더 성능**: 전체 셀 재렌더는 비용↑ → diff 기반 부분 갱신 + push coalescing.
- **Textual 내부 좌표 배치**: 프레임워크 레이아웃과 충돌 가능 → 서버 권위 layout 을 절대 좌표로 두고 클라이언트는
  배치만 따름.
- **데몬 좀비/고아 프로세스**: 패널 종료 시 `waitpid` 회수, 서버 종료 시 자식 정리(SIGTERM→SIGKILL).
- **Windows 미지원**: POSIX PTY 전제. ConPTY 이식은 별도 과제.

## 12. 향후 확장 (비목표/후속)

- 스크롤백 검색(`/` 검색·하이라이트)·텍스트 선택 복사(copy)·페이스트 버퍼 — 기본 스크롤백(R6)은 1차 범위,
  검색/복사는 후속
- 레이아웃 프리셋(tiled/even-h/even-v)
- 세션 레이아웃 직렬화(재부팅 후 부분 복원)
- 설정 파일(`~/.config/pytmux/config`)로 키맵·색상·상태줄 포맷 커스터마이즈
- SSH 를 통한 원격 attach
